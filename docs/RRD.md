# 모임 이벤트 관리 웹 MVP PRD

## 🎯 핵심 정보

**목적**: 정기 모임(수영/헬스/친구 모임 등) 주최자가 카톡방·엑셀·수기 계산에 의존해 반복 처리하던 공지, 참여자 관리(RSVP), 정산 업무를 하나의 링크로 대체한다.
**사용자**: 정기 모임을 반복적으로 여는 소규모 주최자(운영자)가 Primary, 링크로 초대받아 참여 여부만 표시하는 참여자가 Secondary.

## 📖 문제 정의

### Before (현재)

1. 주최자가 카톡방에 모임 공지를 텍스트로 작성해 올린다.
2. 참여자들이 대화창에 "저요", "불참합니다" 등을 흩어진 메시지로 남긴다.
3. 주최자가 대화 내용을 스크롤하며 참여 인원을 수기로 엑셀/메모에 정리한다.
4. 모임 후 총비용을 인원수로 나눠 계산기로 1인당 금액을 구하고, 누가 입금했는지 다시 카톡으로 확인하며 체크한다.
5. 다음 모임 때 이 과정을 처음부터 반복한다.

### After (도입 후)

1. 주최자가 로그인 후 이벤트(제목/일시/장소/공지사항)를 생성하면 공유 링크가 자동 발급된다.
2. 주최자가 카톡방 등 기존 채널에 링크만 공유한다.
3. 참여자는 로그인 없이 링크 접속 → 이름 입력 → 참여/불참 선택만으로 RSVP를 완료한다.
4. 주최자는 대시보드에서 실시간 참여 인원을 확인하고, 모임 후 총비용을 입력하면 1인당 금액이 자동 계산되어 공개 페이지에 노출된다.
5. 주최자는 입금 여부만 체크하고, 참여자는 같은 공개 페이지에서 정산 내역을 스스로 확인한다.

## 👥 타겟 사용자 & 페르소나

### Primary — 주최자 (운영자)

- 수영/헬스 모임, 스터디, 친구 모임 등을 **정기적으로(주 1회~월 1회 이상)** 여는 사람
- 매번 비슷한 정보(일시/장소/공지)를 반복 입력하고, 인원 파악과 정산을 직접 계산해온 경험이 있음
- Supabase Auth 기반으로 **로그인이 필요**하며, 본인이 만든 이벤트 목록/이력을 관리해야 함

### Secondary — 참여자

- 주최자가 공유한 링크를 받은 사람 (카톡방 멤버 등)
- **회원가입 없이** 이름과 선택적 연락처만 입력해 참여 의사를 표시
- 정산 결과는 같은 공개 링크에서 별도 인증 없이 확인

### Operator — 관리자 (운영자)

- 서비스 전체를 운영·모니터링하는 **시스템 운영자**
- **주최자(호스트)와 완전히 별개 역할** — 주최자용 인증(F011, Supabase Auth)과 구분되는 별도 관리자 로그인 개념
- 전체 이벤트/주최자(회원) 현황 조회, 통계 확인
- 관리 대상: **주최자(회원)** 목록 (참여자/RSVP는 비회원이라 관리 대상 아님)
- **이번 MVP 범위**: 정적 UI 목업만 (실제 관리자 인증/권한 체계, 실데이터 연동, 실통계 API는 제외 — 향후 확장)

## 🚶 핵심 사용자 흐름 (User Flow)

### 주최자 흐름

```
1. 로그인 페이지
   ↓ 로그인 성공 (회원가입은 최초 1회)

2. 이벤트 목록 페이지
   ↓

   [새 이벤트 만들기 클릭] → 3-A. 이벤트 생성 페이지 (빈 입력폼)
   [지난 이벤트 "복제" 클릭] → 3-B. 이벤트 생성/수정 페이지 (제목/장소/공지사항/참가비 안내 자동 채움, 일시·정원은 빈 값)
   ↓

3. 이벤트 생성/수정 페이지
   ↓ 제목/일시/장소/정원(선택)/공지사항/참가비 안내 입력(또는 복제된 값 확인/수정) 후 저장
   ↓ 공유 링크(slug) 자동 생성

4. 이벤트 관리 대시보드
   ↓ 공유 링크를 카톡 등 외부 채널에 복사/공유
   ↓ (시간 경과) 참여자들의 RSVP가 실시간으로 쌓임
   ↓ 참여 인원 확인, 정원 초과 시 경고 확인

5. 모임 종료 후, 같은 대시보드에서 총비용 입력
   ↓ 1인당 금액 자동 계산 → 공개 페이지에 정산 요약 노출
   ↓ 참여자별 입금 여부를 하나씩 수동 체크

6. 완료 → 이벤트 목록으로 복귀, 다음 이벤트 새로 생성 또는 지난 이벤트 복제(반복)
```

### 참여자 흐름

```
1. 카톡 등 외부 채널에서 공유 링크 수신
   ↓ 링크 클릭 (로그인 불필요)

2. 이벤트 공개 페이지 진입
   ↓ 공지사항(제목/일시/장소/공지/참가비 안내) 확인

   [아직 RSVP 안 함] → 이름 + 선택적 연락처 입력 → 참여/불참 선택 → 제출
   [이미 RSVP 함]    → 본인이 남긴 참여 여부 확인 (같은 페이지 재방문 시 노출)
   ↓

3. (모임 이후 재방문 시) 같은 공개 페이지에서 정산 요약 확인
   총비용 / 확정 인원 / 1인당 금액 / 본인 입금 여부(주최자가 체크한 결과) 열람
```

### 스키마 변경 여부 (지난 이벤트 복제 관련)

지난 이벤트 복제(F012)는 기존 `events` row에서 `title`, `location`, `notice`, `fee_info` 값만 읽어 새 `events` row로 INSERT하고, `slug`는 새로 발급하며 `event_at`/`capacity`/`total_cost`는 비운 채(`created_at`도 새로 생성) 저장하는 애플리케이션 로직만으로 구현 가능하다. 새로 생성된 이벤트의 `rsvps`는 처음부터 빈 상태로 시작하므로 별도 초기화 로직도 필요 없다. 따라서 **`events`, `rsvps` 테이블에 컬럼 추가나 스키마 변경은 필요하지 않다.** ("원본 이벤트 참조" 필드는 복제 이력 추적이나 "원본 보기" 같은 부가 기능에서나 의미가 있는데, 이번 MVP 범위에는 해당 요구사항이 없으므로 추가하지 않는다.)

### 스키마 변경 여부 (관리자페이지 관련)

관리자페이지(F013~F017)는 이번 MVP에서 **정적 UI 목업만** 다루며 모든 화면이 더미 데이터로 렌더된다. 따라서 **DB 스키마 변경(테이블/컬럼 추가, RLS 정책)은 필요하지 않다.** 관리자 전용 인증/권한 체계, 관리자용 실데이터 조회, 실통계 집계는 향후 확장 단계에서 별도 설계한다.

- **npm** - 의존성 관리

## 🚫 MVP 제외 범위 (Out of Scope)

| 항목                       | 제외 사유                                                                   |
| -------------------------- | --------------------------------------------------------------------------- |
| 카풀/이동 조율             | 핵심 3대 기능(공지/RSVP/정산)에 집중, 별도 매칭 로직 복잡도가 큼            |
| 실제 결제/송금(PG) 연동    | 소규모 모임 정산은 수동 계좌이체로 충분, PG 연동은 개발 비용 대비 효용 낮음 |
| 참여자 회원가입/로그인     | 참여자는 링크 접근만으로 충분, 회원가입은 진입 장벽만 높임                  |
| 반복 모임 그룹/정기 알림   | 매번 개별 이벤트 생성 구조로 MVP 검증, 그룹화는 사용 패턴 검증 후 결정      |
| 참여자 개인별 비공개 확인  | 공개 페이지 하나로 투명하게 노출하는 방식이 확정 정책                       |
| 대기열/자동 정원 관리      | 정원 초과는 경고 표시로만 대응, 자동 대기열은 운영 복잡도 증가              |
| 관리자 실제 인증/권한 체계 | 이번 관리자페이지는 정적 UI 목업만, 별도 운영자 인증/RBAC는 향후 설계       |
| 관리자용 실데이터 연동     | 관리자 대시보드/테이블은 더미 데이터로만 구현, 실 DB 조회 연동은 향후 확장  |
| 실통계 API 연동            | 통계 분석은 Recharts 더미 차트만, 실통계 집계 API는 향후 확장               |

## 📊 성공 지표

- **이벤트 생성 후 실제 RSVP 발생 여부**: 생성된 이벤트 중 1건 이상 RSVP가 제출된 이벤트의 비율
- **정산 기능 사용률**: 종료된 이벤트 중 총비용 입력 및 정산 계산이 실제로 사용된 비율
- **주최자 재사용률**: 이벤트를 2회 이상 생성한 주최자의 비율 (반복 사용 여부로 핵심 가치 검증)

## 🔮 향후 확장 아이디어 (Post-MVP)

- 카풀 매칭 (참여자 간 이동 수단 조율)
- 실제 송금 연동 (계좌 자동 이체, PG 결제)
- 반복 모임 그룹화 및 정기 알림(예: "수요 필라테스 모임" 그룹 단위 관리)
- 참여자 계정화 (로그인 기반 개인별 참여 이력 관리)
- 대기열 및 자동 정원 관리
